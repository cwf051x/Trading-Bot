# AI Review Workflow

本项目采用「Codex 实现 + GPT 独立审查 + GitHub PR 固化」的代码修改流程。

目标不是让 Codex 写完代码就直接上线，而是建立一个稳定、可追溯、可返工的开发闭环。

开始新任务前，Codex 应先阅读 `PROJECT_CONTEXT.md`、`AGENTS.md`、`README.md` 和本文件；其中 `PROJECT_CONTEXT.md` 负责当前项目稳定上下文，`AGENTS.md` 负责硬规则，本文件只负责 PR 审查闭环。

```text
用户提出需求
  ↓
Codex 在独立分支实现
  ↓
Codex 提交 commit / push / PR
  ↓
Codex 生成给 GPT 的审查提示词
  ↓
GPT 审查 PR / diff / 测试结果
  ↓
GPT 输出审查报告 + 给 Codex 的返工提示词
  ↓
Codex 在同一个 PR 分支返工
  ↓
GPT 二审
  ↓
用户决定是否 merge / deploy
```

---

## 一、角色分工

### 1. 用户

用户是最终负责人，负责：

1. 提出需求。
2. 判断功能是否符合预期。
3. 决定是否合并 PR。
4. 决定是否部署生产环境。
5. 对涉及资金、交易、服务器、安全配置的改动做最终确认。

### 2. Codex

Codex 是代码执行者，负责：

1. 阅读现有代码。
2. 在独立分支中实现需求。
3. 保持最小可行修改。
4. 提交 commit。
5. 创建或更新 Pull Request。
6. 运行必要测试。
7. 生成给 GPT 的审查提示词。
8. 根据 GPT 审查意见返工。
9. 返工后追加 commit 到同一个 PR，不要重开无关 PR。

### 3. GPT

GPT 是独立审查者，负责：

1. 审查 Codex 的 PR / diff / commit / 测试结果。
2. 判断需求是否真正完成。
3. 检查潜在 bug、破坏性变更、安全风险、交易逻辑风险。
4. 输出审查报告。
5. 生成可直接复制给 Codex 的返工提示词。
6. 给出人工验收命令。

---

## 二、Codex 工作硬规则

Codex 每次修改代码必须遵守：

1. 禁止直接修改 `main` / `master`。
2. 必须在独立分支中完成修改。
3. 禁止未经用户确认直接部署生产环境。
4. 禁止未经用户确认合并 PR。
5. 禁止为了通过测试而删除测试、降低校验标准、绕过错误处理。
6. 禁止擅自扩大需求范围。
7. 禁止隐藏未解决风险。
8. 涉及交易、下单、资金、API key、token、缓存、定时任务、并发、数据库、Docker、Nginx、端口、防火墙、证书的修改必须保守处理。
9. 如果需求不清晰，优先做最小可行方案，并在输出中明确假设。
10. 每次完成后必须输出给 GPT 的审查提示词。

## 三、Codex 完成开发后必须输出的内容

Codex 完成一轮开发后，必须输出以下内容：

```text
## 本次需求复述

用自己的话复述用户需求。

## 分支信息

- 当前分支：
- 基准分支：
- PR 链接：

## 修改摘要

说明本次做了哪些修改。

## 修改文件列表

列出主要修改文件，并说明每个文件的作用。

## 核心逻辑变化

说明关键代码逻辑、数据流、调用链变化。

## 已运行测试

列出运行过的命令和结果，例如：

- git status
- git diff main...HEAD
- python -m compileall .
- pytest -q
- ruff check .
- mypy .
- npm run lint
- npm run typecheck
- npm run build
- docker compose config
- docker compose ps
- docker compose logs --tail=100

## 未验证风险

说明哪些地方尚未验证，不能假装已经验证。

## 可能影响的旧功能

说明本次修改可能影响哪些旧功能。

## 回滚方案

说明如何回滚本次修改。

## 给 GPT 的审查提示词

生成一段可以直接复制给 GPT 的审查提示词。
```

## 四、Codex 生成给 GPT 的审查提示词模板

Codex 完成 PR 后，请自动生成以下提示词。

```text
你现在是我的独立代码审查员，请严格审查 Codex 刚完成的 PR / diff / commit。

项目背景：
- 项目名称：
- 运行环境：
- 主要技术栈：
- 是否涉及交易 / 下单 / 资金 / API key / 服务器部署：
- 本次原始需求：

PR 信息：
- 仓库：
- 分支：
- PR 链接：
- 基准分支：

Codex 修改摘要：
粘贴 Codex 的修改摘要。

修改文件：
粘贴主要修改文件列表。

核心逻辑变化：
粘贴 Codex 对核心逻辑变化的说明。

测试命令和结果：
粘贴已运行的测试命令和结果。

未验证风险：
粘贴 Codex 声明的未验证风险。

可能影响的旧功能：
粘贴 Codex 声明的影响范围。

请重点审查：

1. 是否真正完成原始需求。
2. 是否存在需求理解偏差。
3. 是否引入破坏旧功能的风险。
4. 是否有交易逻辑风险，例如重复下单、错误开仓、错误止损、错误仓位、dry-run/live 混淆。
5. 是否有异步、缓存、定时任务、API 限频、WebSocket/REST 数据一致性问题。
6. 是否有安全风险，例如 API key 泄露、token 泄露、端口暴露、权限过大、生产环境配置错误。
7. 是否有异常处理遗漏。
8. 是否有性能问题。
9. 是否有测试不足。
10. 是否存在为了完成需求而过度重构的问题。
11. 是否建议合并。

请按以下格式输出：

一、总体结论：
可合并 / 有条件合并 / 不建议合并

二、高风险问题：
列出必须修复的问题。每个问题说明：
- 问题描述
- 影响范围
- 风险等级
- 建议修复方式

三、中风险问题：
列出建议修复的问题。

四、低风险问题：
列出代码风格、可维护性、边界情况等问题。

五、必须让 Codex 返工的事项：
按优先级列出。

六、可直接复制给 Codex 的返工提示词：
请生成一段完整提示词，让 Codex 在当前 PR 分支继续修复，不要新建分支，不要合并，不要部署。

七、人工验收命令：
列出用户在本地或服务器上应该执行的检查命令。

八、最终建议：
说明本 PR 当前是否可以合并，以及合并前还需要完成什么。
```

## 八、人工验收基础命令

根据项目类型选择运行。

### 1. Git 检查

```bash
git status
git branch --show-current
git log --oneline -5
git diff main...HEAD
```

### 2. Python 项目

```bash
python -m compileall .
pytest -q
ruff check .
mypy .
```

### 3. Node / 前端项目

```bash
npm install
npm run lint
npm run typecheck
npm run build
```

### 4. Docker 项目

```bash
docker compose config
docker compose ps
docker compose logs --tail=100
```

### 5. 服务器部署相关

```bash
systemctl status nginx --no-pager
nginx -t
sudo ufw status verbose
docker ps
docker logs --tail=100 <container_name>
```

### 6. 交易系统额外检查

```text
1. 确认 dry-run / live 模式没有混淆。
2. 确认 API key 没有被提交到 Git。
3. 确认不会重复下单。
4. 确认不会重复发通知。
5. 确认定时任务不会重复启动。
6. 确认缓存过期时间合理。
7. 确认接口异常时不会无限重试。
8. 确认资金、仓位、止损、止盈逻辑没有反向或单位错误。
9. 确认日志不会泄露 token、secret、key。
10. 确认生产环境配置没有被测试配置覆盖。
```

## 九、合并前检查清单

合并 PR 前，用户应确认：

```text
[ ] PR 来自独立分支，不是直接改 main。
[ ] Codex 已说明本次需求、修改摘要、测试结果、风险。
[ ] GPT 已完成独立审查。
[ ] 高风险问题已经修复或明确决定暂缓。
[ ] Codex 返工后追加 commit，而不是重开无关分支。
[ ] 二审结论为可合并或有条件合并。
[ ] 用户已运行必要验收命令。
[ ] 没有泄露 API key、token、secret。
[ ] 没有误改生产环境配置。
[ ] 没有直接部署生产环境。
[ ] 用户明确决定 merge。
```

---

## 十、禁止事项

```text
1. 禁止 Codex 直接 merge。
2. 禁止 Codex 直接部署生产环境。
3. 禁止 Codex 删除测试来通过检查。
4. 禁止 Codex 擅自降低风控、异常处理、权限校验。
5. 禁止 Codex 把 API key、token、secret 写入代码或日志。
6. 禁止 Codex 把审查意见当作“可选建议”而不回应。
7. 禁止 GPT 只做泛泛审查，必须输出具体问题和返工提示词。
8. 禁止用户在未看测试结果、未看 GPT 审查结论的情况下合并高风险 PR。
```
